# DittoBench Wire Protocol (SN118)

This is the shared contract between the DittoBench **validator** and your
**miner harness**. The Go source of truth is
[`pkg/protocol/protocol.go`](pkg/protocol/protocol.go); these types must stay
byte-compatible with the `dittobench-api`.

There are three artifacts:

1. **Dataset** — a seeded set of tool-calling cases (generated by the
   validator; the practice generator in `pkg/datagen` mimics it).
2. **/run request + response** — the per-case call the validator makes to your
   harness HTTP server.
3. **Score report** — the result of scoring a run.

---

## 1. Dataset

```json
{
  "seed": 1718500000,
  "generated_at": "2026-06-16T00:00:00Z",
  "tool_cases": [
    {
      "id": "web_search-0003",
      "category": "web_search",
      "prompt": "What's the latest news on GraphQL?",
      "expected_tools": [
        { "name": "search_web", "required_args": { "query": "" }, "forbidden_args": [] }
      ],
      "max_tool_calls": 2,
      "allow_extra_tools": false,
      "expected_behavior": "Should call search_web for fresh information."
    }
  ]
}
```

| Field | Type | Notes |
|-------|------|-------|
| `seed` | int64 | Deterministic seed. |
| `tool_cases[].id` | string | Unique case id. |
| `tool_cases[].category` | string | e.g. `web_search`, `memory_lookup`, `abstention`. |
| `tool_cases[].prompt` | string | The user input. |
| `tool_cases[].expected_tools` | []ToolSpec | Tools the case expects (empty = abstain). |
| `tool_cases[].max_tool_calls` | int | Soft cap on calls. |
| `tool_cases[].allow_extra_tools` | bool | If false, extra calls are penalized. |

**ToolSpec**

| Field | Type | Notes |
|-------|------|-------|
| `name` | string | Tool name. |
| `required_args` | map[string]string | Args expected (validator-side checks). |
| `forbidden_args` | []string | Args that must not appear. |

---

## 2. `/run` Request → Response

The validator POSTs one `RunRequest` per case to your harness at `POST /run`.

### Request

```json
{
  "case_id": "web_search-0003",
  "system_prompt": "You are Ditto ...",
  "user_input": "What's the latest news on GraphQL?",
  "tools": [
    {
      "name": "search_web",
      "description": "Search the public web for fresh information.",
      "parameters": { "type": "object", "properties": { "query": { "type": "string" } }, "required": ["query"] }
    }
  ]
}
```

`tools[].parameters` is a JSON-Schema object (passed straight through to the
model's tool/function definitions).

### Response

Your harness returns a `RunResponse`:

```json
{
  "final_text": "Here is what I found ...",
  "tool_calls": [
    { "name": "search_web", "args": { "query": "GraphQL news" }, "hop": 0 }
  ],
  "prompt_tokens": 512,
  "output_tokens": 88,
  "latency_ms": 740
}
```

| Field | Type | Notes |
|-------|------|-------|
| `final_text` | string | Final assistant answer. |
| `tool_calls` | []ObservedToolCall | Every tool call made, in order. |
| `tool_calls[].name` | string | Tool name. |
| `tool_calls[].args` | JSON | Raw args object. |
| `tool_calls[].hop` | int | Which tool-calling round (0-based). |
| `prompt_tokens` / `output_tokens` | int64 | Token usage (feeds cost scoring). |
| `latency_ms` | int64 | Wall-clock for the case (feeds speed scoring). |

Also expose `GET /health` → `{"status":"ok"}`.

## 3. Score Report

```json
{
  "run_id": "practice-1718500000",
  "generated_at": "2026-06-16T00:00:00Z",
  "composite": 0.86,
  "tool_mean": 0.86,
  "median_ms": 720,
  "n": 20,
  "per_case": [
    {
      "case_id": "web_search-0003",
      "category": "web_search",
      "tool_score": 1.0,
      "latency_ms": 740,
      "called": ["search_web"],
      "expected": ["search_web"],
      "notes": []
    }
  ]
}
```

### Tool-accuracy scoring (per case)

Mirrors the validator (`pkg/scorer`):

- For each expected tool, `matched = min(expected_count, observed_count)`.
- `base = sum(matched) / total_expected_count`.
- If `allow_extra_tools == false`, subtract `0.1` per unexpected extra call.
- Clamp to `[0, 1]`.
- **Abstention cases** (no expected tools) score `1.0` only if the harness
  called **no** tools; otherwise `0.0`.

`composite` / `tool_mean` = mean of per-case `tool_score`. `median_ms` is the
median per-case latency.

> The on-chain composite additionally folds in **token cost** and **wall-clock**
> (and, on the validator only, **memory QA**). Offline practice scores
> tool-accuracy + reports latency/tokens so you can iterate cheaply.
